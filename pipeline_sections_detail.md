# 파이프라인 보고서 상세 설명 (HTML 섹션별)

> `pipeline_overview.html`의 요약을 바탕으로 한 확장 설명입니다.  
> 현재 작성 범위: **섹션 1~2**. 섹션 3~12는 이후 확장 예정.

## 1. 한눈에 보기 (HTML §1)

### 1.1 이 섹션이 하는 일

보고서 첫 부분은 **전체 프로젝트의 “규모와 구성 요소”를 숫자 카드**로 압축해 보여 줍니다. 코드 저장소를 처음 여는 사람이 “무엇이 얼마나 들어 있는 파이프라인인지”를 1분 안에 파악하도록 설계된 **대시보드형 요약**입니다.

### 1.2 카드별 의미

| 표시 | 의미 (파이프라인 관점) |
|------|------------------------|
| **11 데이터 소스** | TCGA-BRCA, METABRIC, DrugBank, ChEMBL, LINCS, STRING, MSigDB, GDSC, Open Targets, DepMap, (선택) PubMed 등 **서로 다른 기관·형식의 원천 데이터**가 11갈래로 들어옵니다. 각각 유전자 발현, 약물–타겟, PPI, 약물 감수성 라벨 등 **역할이 다르므로** “한 데이터셋”이 아니라 **다소스 통합**이 핵심입니다. |
| **56 Snakemake Rules** | 전처리를 **워크플로 도구 Snakemake**의 규칙(rule) 단위로 쪼개어 자동 실행한다는 뜻입니다. Rule 수가 많다는 것은 **단계가 세분화**되어 있고(로드, 필터, QC, 변환, 저장 등), 재실행·병렬화·의존성 그래프 관리가 가능하다는 신호입니다. |
| **10 ML 데이터셋** | 전처리 **중간·최종 산출물 14개**를 `integrate.py`에서 **ML에 맞게 통합·정렬**해 **10개의 “학습/분석용 테이블(패킷)”**으로 만든다는 요약입니다. (실제 컬럼·shape는 HTML §5 표 참고.) |
| **12 ML 모델** | 최종 예측·랭킹을 **하나의 모델에만 맡기지 않고**, 전통 ML·딥러닝·지식 기반 점수를 **3개 트랙(Track A/B/C)**으로 나누어 총 12개 모듈을 둔다는 의미입니다. |
| **~2,064 피처** | (약물, 세포주) 단위로 모델에 들어가는 입력 차원의 대략치입니다. HTML §6에 따르면 **약물 쪽 ~2,011**(타겟 유전자 이진 벡터가 대부분) + **세포주 쪽 53**(DepMap CRISPR·PRISM 등)으로 구성됩니다. |
| **7,200+ 학습 샘플** | (drug, cell line) 쌍 등 **라벨이 붙은 학습 단위**의 개수입니다. “매칭 보강 후 증가 예상”은 **약물명 정규화·매칭(`drug_harmonize.py` 등)**으로 쓸 수 있는 쌍이 늘어날 수 있다는 전제를 적어 둔 것입니다. |
| **14 QC3 체크포인트** | 학습·앙상블 이후 **품질 검증(validate.py)**을 수치 기준과 함께 통과해야 한다는 **게이트**가 14개 있다는 뜻입니다. (HTML §10, AUROC·누수·LODO/LOCO 등.) |
| **14 전처리 모듈** | 데이터 소스/주제별로 **전처리 파이프라인이 모듈화**되어 있다는 요약 표현으로 읽을 수 있습니다. (세부 Rule 번호·파일명은 HTML §4.) |

### 1.3 환경 한 줄의 의미

> `SageMaker(ml.m5.xlarge)`에서 `Snakemake` 실행(전처리) → `Python scripts`(피처) → 팀원 `AWS Glue/EMR`(대규모) → `SageMaker(ml.g4dn.12xlarge)`(학습 예정)

- **m5.xlarge**: CPU 위주 인스턴스로 **I/O·병렬 전처리(Snakemake)**에 적합한 선택으로 해석할 수 있습니다.
- **Snakemake**: 전처리 단계를 **재현 가능한 파이프라인**으로 묶는 역할입니다.
- **Python 스크립트**: 통합(`integrate.py`)·피처(`features.py`) 등 **ML 직전 로직**이 여기에 해당합니다.
- **Glue/EMR**: 팀 단위 **대규모 ETL/집계**가 별도 트랙으로 있다는 점을 명시합니다(규모·피처 수가 다름 — HTML §8, §11).
- **g4dn.12xlarge**: GPU(T4)가 있는 인스턴스로 **딥러닝 학습(TabNet, GNN 등)**을 예상한 단계입니다.

### 1.4 섹션 1에서 “요약 너머”로 알아두면 좋은 점

- 숫자는 **설계 목표·현재 스냅샷**일 수 있어, 실제 S3의 parquet 행 수·컬럼 수와 **반드시 1:1로 같지 않을 수** 있습니다. 최종은 **실행 로그·데이터 프로파일**로 맞추는 것이 좋습니다.
- **11 소스 ≠ 11개 파일**: 한 소스 안에도 여러 파일·버전이 있을 수 있습니다.
- **12 모델**은 운영 시 **모두 매번 학습**하기보다, 단계적 도입·앙상블 가중 조정이 현실적입니다.

## 2. 전체 파이프라인 흐름 (HTML §2)

### 2.1 이 섹션이 하는 일

섹션 2는 **데이터가 위에서 아래로 흐를 때 “각 단계에서 얼마나 걸러지고, 무엇이 남는지”**를 **숫자와 짧은 문장**으로 순서도에 넣은 부분입니다. 즉 **“왜 이 필터인가?”**를 한 줄씩이라도 남기려는 **방법론 설명용 플로우**입니다.

### 2.2 단계별 상세 (위→아래)

#### (1) 입력: 11개 데이터 소스

- TCGA, METABRIC, DrugBank, ChEMBL, GDSC, STRING, MSigDB, LINCS, Open Targets, DepMap 등이 **한 번에 들어오는 그림**입니다.
- 여기서는 아직 **통합 행렬 하나**가 아니라, **소스별 원천 규모**가 강조됩니다(예: TCGA 60,660 genes, STRING ~1.4M edges).

#### (2) QC1: Raw Schema Validation (10개 소스)

- **“스키마 검증”**은 컬럼명·타입·필수 필드·결측 한도 등 **원본이 약속한 형식을 지키는지** 먼저 확인하는 단계입니다.
- 11개 중 **10개**를 명시한 것은 PubMed 등 **optional 소스**를 QC1 범위에서 다르게 둔 것과 맞물립니다(HTML §3 표).

#### (3) 전처리: Snakemake 56 rules (SageMaker m5.xlarge)

HTML에 적힌 **대표적인 축소 숫자**는 다음과 같은 **생물학·데이터 품질 논리**를 압축한 것입니다.

- **protein_coding 필터 (60,660 → ~20,000 유전자)**  
  비단백질 코딩 RNA 등은 **약물 타겟 후보로 쓰기 어렵다**는 전제에서 **분석 공간을 단백질 코딩 유전자로 제한**합니다.
- **low expression 필터 (~20,000 → ~15,000)**  
  표본 대다수에서 신호가 거의 없는 유전자는 **노이즈·불안정성**이 커서 제거합니다.
- **PPI score ≥ 700 (1.4M → 380K 엣지)**  
  STRING의 combined score 기준으로 **낮은 신뢰 엣지**를 줄여 **그래프 기반 모델·근접도 분석의 신뢰도**를 올립니다.
- **GDSC breast cancer (1M → 12K IC50)**  
  **유방암 세포주(및 메타데이터 기준)**에 맞춰 감수성 라벨 공간을 맞춥니다. “전 암종”이 아니라 **BRCA 과제**에 초점을 맞춘 결과입니다.
- **withdrawn/illicit 약물 제거 (~2–3%)**  
  안전성·규제 이슈가 있는 약물을 **재창출 후보에서 제외**합니다.

이 단계의 산출은 **소스별 정제된 테이블·중간 parquet**들이며, 이후 통합의 재료가 됩니다.

#### (4) QC2: Pandera 35 + Final 16-point

- **Pandera**는 DataFrame에 대해 **스키마·값 범위·제약**을 코드로 선언해 검증하는 도구입니다. “35개”는 **세부 검증 항목·스키마 블록 수**로 이해하면 됩니다.
- **Final 16-point**는 전처리 후단의 **최종 체크리스트**(결측, 중복, 키 일관성, 분포 이상 등)로, HTML §4·§5의 **병합 키(hugo_symbol 등)**가 여기서 깨지지 않았는지 확인하는 성격입니다.

#### (5) ML 통합: `integrate.py` (14 산출물 → 10 ML-ready parquet)

- **14개 전처리 산출물**을 **같은 ID 체계**(HUGO, UniProt, drug id 등)로 맞춘 뒤, ML이 소비하기 쉬운 **10개 테이블**로 재구성합니다.
- HTML에 나온 예:
  - **TCGA ∩ METABRIC 공통 유전자 ~14,000** — 플랫폼이 달라도 **같은 유전자 축**으로 발현을 맞추기 위함입니다.
  - **DrugBank + ChEMBL 통합 ~140K drug–target pairs** — 구조화된 타겟 정보와 **활성(pChEMBL)** 정보를 한 흐름으로 쓰기 위함입니다.

#### (6) 피처 엔지니어링: `features.py`

- **Drug-level ~2,011**: 타겟 유전자 이진(대부분) + pChEMBL 요약 + ADMET 게이트 + pathway overlap + 타겟 발현 요약 등.
- **Cell-line-level 53**: DepMap CRISPR top-50 + PRISM 요약 등으로 **같은 약물이라도 세포주별 이질성**을 잡습니다.
- **약물 매칭 26% → 90%+**: 서로 다른 DB의 약물명을 **정규화·계층 매칭**으로 맞추는 과정으로, **라벨과 피처를 같은 약물 키로 붙이기** 위한 핵심입니다.

#### (7) 12 모델 학습 (3-Track)

- **Track A**: ElasticNet, RF, XGBoost, LightGBM, MLP, TabNet 등 **표 형식 피처**에 강한 모델군, nested CV 등으로 **일반화**를 노립니다.
- **Track B**: AE, VAE(생물학 제약 옵션), GNN(GATv2) 등 **표현 학습·그래프** 기반.
- **Track C**: Signature reversal, network proximity, pathway overlap 등 **도메인 지식·통계** 기반 점수.

#### (8) 앙상블: RRF 7-ranking

- 서로 다른 스케일의 점수를 **순위 기반으로 합치는** RRF(Reciprocal Rank Fusion) 계열 접근입니다. Track A/B/C가 **각각 약물 랭킹**을 만들고 이를 합칩니다.

#### (9) QC3: 14-point + 최종 랭킹 + 에이전트 + QC4

- **QC3**는 모델·앙상블 결과가 **최소 품질 기준**을 만족하는지 봅니다(LODO/LOCO 등 외적 타당성 포함).
- 이후 **Claude + MCP 에이전트**로 문헌·설명 보강을 시도하고, **QC4 Guardrails**로 안전·정책 층을 둔다는 **운영 상의 마지막 방어선**으로 서술되어 있습니다.

### 2.3 섹션 2를 읽을 때의 주의점

- 화살표 사이 숫자는 **대표값·반올림**이므로, 논문·재현 패키지용으로는 **실행 로그의 정확한 n**이 필요합니다.
- **Glue/EMR 팀 파이프라인**과 **본 파이프라인**은 표본 수·피처 수가 다릅니다(HTML §8, §11). 같은 “팀 과제”라도 **산출물 정의가 다를 수 있음**을 염두에 두어야 합니다.

## 계획 보강 (HTML·슬라이드 대비) — METABRIC, ADMET, Bedrock, Nextflow, Batch

`pipeline_overview.html`은 전처리·ML 통합·피처·모델·앙상블·QC3까지를 잘 요약하지만, **주차별 슬라이드에 있는 아래 단계가 “워크플로 한 줄”로는 빠져 있거나 약함**으로 정리할 수 있다.

### A. METABRIC 단계 (외부 코호트 일반화)

- **의도:** TCGA(또는 동일 분포에 가까운 훈련 데이터)에서 만든 모델·랭킹이 **독립 코호트(METABRIC)**에서도 의미 있는지 확인한다.
- **HTML과의 관계:** §10에 External AUROC(METABRIC) 등 **지표는 등장**하지만, §2 플로우차트에는 **“METABRIC 검증 퍼널”이 별도 박스로 분리**되어 있지 않다. 문서·다이어그램에 **Stage 3: 일반화 검증**을 명시하는 것이 좋다.

### B. ADMET 단계 (임상 진입 전 생물학적 필터)

- **의도:** 후보 약물을 **용해도, CYP 대사, hERG, 간독성** 등 **4 Gate**로 거른 뒤에만 “최종 후보 리스트”에 올린다(Week 4 Part A 슬라이드와 동일 개념).
- **HTML과의 관계:** 피처·후처리에 ADMET 관련 컬럼·게이트가 있으나, **“임상 진입용 필터 단계”**로 이름이 박스에 박혀 있지는 않다. **Bedrock 이전에 반드시 거치는 게이트**로 표기하는 것이 Pre-Project 산출물 정의와 맞다.

### C. Bedrock(LLM) — ADMET·METABRIC 이후의 검증·XAI

- **의도:** 모델 점수·필터 결과를 **문헌/지식 그래프 RAG**와 결합해 근거 검색, 기전 설명, 리스크, **모순 탐지**까지 포함한 **임상 문서형 산출**을 만든다(Week 4 Part B).
- **순서:** “랭킹만 하고 바로 LLM”이 아니라, **일반화(ADMET·METABRIC)로 숏리스트가 정제된 뒤** Bedrock에 넣는 흐름이 슬라이드상 목표와 일치한다.

### D. S3 → Nextflow → (파일럿) ML/DL/Graph → 확장·랭킹

- 입력: 팀 버킷의 **raw + `results/`**(전처리 완료).
- **Nextflow**로 피처 엔지니어링을 정의·재현.
- **소수 산출물·부분 샘플**로 ML/DL/Graph를 먼저 스모크 테스트한 뒤, **전체 확장 → 모델별 랭킹 → 최고 모델**로 METABRIC·ADMET·Bedrock 후속에 연결.

### E. AWS Batch

- 대량 파일·독립 작업 병렬에 적합. Nextflow `awsbatch` 실행자와 궁합이 좋다.
- GPU 작업은 **별도 큐/컴퓨트 환경** 분리, S3 I/O 반복이 크면 **스테이징(FSx 등)** 검토. 상세 의견은 저장소 **`README.md`의 “AWS Batch로 돌릴지에 대한 의견”** 절 참고.
